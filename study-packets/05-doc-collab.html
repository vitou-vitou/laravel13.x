Step 1 — Understanding
Principle 1 — Map of the System

How a block-based editor fits together

A Notion-style app is a tree of typed blocks, edited concurrently, that must merge without losing work. The pressure: a document is not a string but a nested structure of blocks (paragraph, heading, list, table, embed); many people type at once; and every change should sync live and be recoverable.

Workspace
 └─ Page (a document, itself a block)
     └─ Blocks (ordered tree)
         ├─ Heading
         ├─ Paragraph
         ├─ Toggle ─► child blocks (nesting)
         ├─ List item ─► child blocks
         └─ Embed / Table / Image
   + Permissions (page-level share)
   + Versions (snapshots / op log)

The defining action — editing a block — persists a granular change and syncs it:

User edits block ──► PATCH /blocks/{id} {content, op}
   │ authorize (can edit page?)
   ▼
 Apply change to block (content JSON) — granular, not whole-page
   │
   ├─► broadcast BlockUpdated to page.{id}  (others merge live)
   └─► append to version/op log (queued)
  • Block model — a page is a block; blocks form an ordered, nested tree with typed content.
  • Concurrent editing — granular block-level changes + a conflict strategy (block-locking, OT, or CRDT).
  • Real-time sync — broadcasting per-block changes to all viewers.
  • Sharing & versions — page-level permissions and a recoverable history.
The mental model in one sentence
A block editor is an ordered nested tree of typed blocks edited concurrently. The hard parts are the block tree (ordering + nesting) and merging simultaneous edits without clobbering.
Principle 2 — Clear Explanations

The concepts, defined precisely

Everything is a block

A block has a type (heading/paragraph/list/table/embed), a JSON content, a parent_id (for nesting), and a position (for ordering among siblings). A page is itself a block. This uniform model lets any block contain any block.

Ordering + nesting in one table

Blocks form a tree via parent_id and order among siblings via position (use fractional/lexorank, as in a Kanban board, so inserting a block is one write). Rendering = recursively fetch children ordered by position. Beware deep recursion; cap nesting or load lazily.

Granular edits, not whole-page saves

Saving the entire document on each keystroke is catastrophic for collaboration — two users overwrite each other. Instead persist per-block changes: a PATCH touches one block's content. Smaller writes mean smaller conflict windows.

The concurrency strategy spectrum

Three options, increasing complexity: (1) Block-level locking / last-write-wins — simplest, occasional lost edits within a block; fine for light collaboration. (2) Operational Transformation (OT) — transform concurrent ops against each other; powers Google Docs, complex. (3) CRDTs (e.g. Yjs) — conflict-free merge by construction; increasingly the default for rich editors, with the server as a relay/persistence layer.

Server's role with CRDTs

With a CRDT the heavy merge logic lives client-side; Laravel's job is to relay updates over WebSockets and persist the document state/op log so a new client can load and catch up. The server becomes an authority for permissions and storage, not text merging.

Sharing & permissions

Permissions are per page, often inherited down a page tree: a user/group has a role (view/comment/edit/full). Resolve effective access by walking up to the nearest explicit grant. Public share links are tokens with a scope and optional expiry.

Version history

Recoverability comes from either periodic snapshots of the block tree or replaying the op log to a point in time. Snapshots are simple to restore; op logs are granular but heavier to replay. Many apps snapshot periodically and keep recent ops.

Most common misconception
Saving the whole document per change is the classic block-editor mistake — it guarantees lost edits under collaboration. Persist granular per-block changes and choose an explicit merge strategy.
Principle 3 — Different Media

The same ideas, four ways

One-line summary

A block editor stores documents as an ordered nested tree of typed blocks, persists granular per-block edits, merges concurrent changes via locking/OT/CRDT, and syncs everything live.

Analogy — LEGO instructions edited by a team

A page is a model built from LEGO bricks (blocks); each brick has a type and can have bricks stacked on it (nesting). Editing one brick doesn't require rebuilding the whole model (granular edits). When several people build at once, you need a rule for who places a brick in a contested spot — a lock (one at a time), a referee transforming moves (OT), or magic bricks that always click together regardless of order (CRDT). A snapshot is photographing the build so you can rebuild it exactly later.

Principle 4 — Short Lessons

Bite-sized micro-lessons

Each lesson is one idea. Code illustrative.

Lesson 1 — One uniform block model

// blocks: id, page_id, parent_id(nullable), type, content(json), position(float)
class Block extends Model {
    protected $casts = ['content'=>'array','position'=>'float'];
    public function children(): HasMany {
        return $this->hasMany(Block::class,'parent_id')->orderBy('position');
    }
}

Lesson 2 — Insert a block with fractional position

$block = Block::create([
    'page_id'=>$page->id,'parent_id'=>$parentId,'type'=>'paragraph',
    'content'=>['text'=>''],
    'position'=>positionBetween($prevPos,$nextPos), // single write, no renumber
]);

Lesson 3 — Patch one block, not the page

public function update(Request $r, Block $block) {
    $this->authorize('edit', $block->page);
    $block->update(['content'=>$r->input('content')]);
    broadcast(new BlockUpdated($block))->toOthers();
    return new BlockResource($block);
}

Lesson 4 — Relay CRDT updates

// server stores the opaque CRDT update and rebroadcasts it
class DocUpdated implements ShouldBroadcast {
    public function __construct(public int $pageId, public string $update) {}
    public function broadcastOn(): PrivateChannel {
        return new PrivateChannel('page.'.$this->pageId);
    }
}
// persist updates (append) for late joiners to replay

Lesson 5 — Resolve inherited permissions

public function effectiveRole(User $u, Page $page): ?string {
    for ($p = $page; $p; $p = $p->parent) {
        if ($grant = $p->grants()->forUser($u)->first()) return $grant->role;
    }
    return null; // no access
}

Lesson 6 — Snapshot for version history

class SnapshotPage implements ShouldQueue {
    public function handle(Page $page): void {
        PageVersion::create([
            'page_id'=>$page->id,
            'tree'=>$page->loadTree()->toArray(), // serialized block tree
            'created_by'=>auth()->id(),
        ]);
    }
} // scheduled or on significant change

Lesson 7 — Public share links are scoped tokens

$link = $page->shareLinks()->create([
    'token'=>Str::random(40),'role'=>'view','expires_at'=>now()->addDays(7),
]);
// middleware resolves token -> grants read-only access to that page subtree
Step 2 — Automaticity
Principle 5 — Test Yourself

Retrieval-practice quiz

Answer before revealing.

Q1. Why model a page and its contents as one uniform block type?
A block has type + content + parent_id + position, so any block can contain any block. Treating a page as a block lets nesting, ordering, and rendering use a single recursive model rather than special cases per content type.
Q2. How is ordering-with-nesting represented in one table?
parent_id gives the tree; position (fractional/lexorank) orders siblings. Rendering recursively fetches children ordered by position; inserting a block is a single write at the midpoint.
Q3. Why persist per-block edits instead of saving the whole document?
Whole-document saves overwrite concurrent edits, guaranteeing lost work. Per-block PATCHes shrink the write and the conflict window, enabling real collaboration.
Q4. Compare last-write-wins, OT, and CRDT.
LWW/lock: simplest, lossy on conflict, fine for light use. OT: transforms concurrent ops, strong but complex (Google Docs). CRDT: conflict-free merge by construction, medium-high complexity, modern default with the server as relay/persistence.
Q5. With a CRDT, what is Laravel's role?
Relay opaque updates over WebSockets and persist the state/op log so late joiners can catch up. The merge logic is client-side; the server owns permissions and storage, not text merging.
Q6. How are effective page permissions resolved when grants are inherited?
Walk up the page tree from the page to the root, returning the role from the nearest explicit grant for the user/group; no grant means no access.
Q7. Two ways to implement version history, and their trade-off?
Periodic snapshots of the block tree (simple to restore, coarser) vs replaying an append-only op log (granular, heavier to reconstruct). Many apps combine periodic snapshots with recent ops.

Code drill — concurrent block edits don't clobber the page (Pest)

it('edits to different blocks both persist', function () {
    $page = Page::factory()->withBlocks(2)->create();
    [$b1,$b2] = $page->blocks;
    patchBlock($b1, ['text'=>'A']);
    patchBlock($b2, ['text'=>'B']); // concurrent-ish, different rows
    expect($b1->fresh()->content['text'])->toBe('A')
        ->and($b2->fresh()->content['text'])->toBe('B');
});

Principle 7 — Mix It Up

Interleaved quiz

Topics shuffled to train choosing the right idea.

M1. Two users typing in the same paragraph keep overwriting each other. Diagnosis and options?
Whole-block last-write-wins with no merge. Options in order of effort: block-level locking, character-level OT, or a CRDT (Yjs) where the server relays/persists opaque updates and clients merge.
M2. Rendering a deeply nested page is slow / stack-overflows. Two fixes?
(1) Cap nesting depth or load child blocks lazily on expand. (2) Fetch the whole tree in one query and build it in memory (avoid per-node N+1 recursion); index (page_id, parent_id, position).
M3. Inserting a block at the top renumbers all siblings. Better approach?
Use fractional/lexorank positions: new position = midpoint of neighbours, a single write. Same technique as Kanban card ordering.
M4. A shared-link recipient can edit though the link was 'view only'. Flaw?
Share token scope not enforced server-side. The token must map to a role (view) and the policy must check it on every mutation; never trust the client to honour read-only.
M5. A user moved to a sub-page they shouldn't access. How should permission have resolved?
By walking up the page tree to the nearest explicit grant; if none grants the user access, deny. Inherited permissions must be resolved server-side per request.
M6. When is a CRDT overkill?
For light, mostly single-editor documents or an MVP. Block-level locking or last-write-wins is far simpler and adequate; adopt CRDT/OT when true simultaneous co-editing of the same text is a core requirement.
Principle 8 — Don't Stop

Overlearning plan

  • Build the editor twice — last-write-wins vs a CRDT (Yjs) relayed by Laravel — and feel where each breaks.
  • Break collaboration on purpose — switch to whole-page saves and watch two editors clobber each other.
  • Write permission tests first: inherited grants, share-link scope, denied sub-pages.
  • Add comments & mentions anchored to specific blocks.
  • Stretch goals: block-level version diffing, offline editing with later sync, slash-command block insertion, and full-text search across pages (Scout).
