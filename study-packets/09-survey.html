Step 1 — Understanding
Principle 1 — Map of the System

How a survey & polling platform fits together

A survey platform is a data-driven instrument that collects answers, enforces participation rules, and aggregates results. The pressure: creators define questions (not developers), so every question type, branch rule, and anonymity setting is data; the same engine must render the instrument, validate responses, prevent duplicate votes, and compute live or exportable aggregates.

Creator (researcher / admin)
 └─ Survey (title, status, anonymity mode, close_at)
     ├─ Questions (type, options, required, branch rules, position)
     ├─ Responses (answers keyed by question id + respondent token)
     └─ Aggregates (counts, percentages, weighted scores)
          └─ dashboards, exports, real-time broadcasts
 Respondent ──► answers questions ──► validated + deduped ──► stored

The defining action — a response — flows through validation, identity/anonymity checks, and aggregation:

Respondent submits answers
   │
   ▼ load survey schema (questions + branch rules + status)
   ▼ resolve visible questions from branch logic
   ▼ build validation rules dynamically from schema
   ▼ check duplicate vote / quota / survey closed
   ▼ validate ; on fail → errors keyed by question
   │ on pass
   ▼ store Response (answers + respondent_token hash)
   ▼ update aggregate counters (sync or queued)
   ▼ broadcast SurveyResultsUpdated (Reverb/Pusher) if live results enabled
  • Question schema — questions as data (type, options, required, branch rules, scoring weights).
  • Response collection — validate against schema, enforce one-vote rules, handle partial vs complete submissions.
  • Aggregation — precomputed counters vs on-demand queries; how anonymity affects storage.
  • Branching & live results — skip logic re-evaluated server-side; cached aggregates pushed over websockets.
The mental model in one sentence
A survey platform is a runtime interpreter of a question schema plus an aggregation pipeline: one schema drives rendering, validation, and reporting. The hard parts are trustworthy server-side branching, duplicate-vote prevention, and fast aggregates without leaking identity.
Principle 2 — Clear Explanations

The concepts, defined precisely

Survey vs poll

Both collect answers from a schema, but they differ in scope and rules. A poll is typically short (one or few questions), often public, with immediate tallies and strict one-vote enforcement. A survey is longer, may be multi-page, supports branching, quotas, and partial completion. In Laravel, model both as the same Survey with a kind flag or shared tables — the engine is identical; UX and validation rules differ (e.g. polls reject partial saves).

Questions are data, not code

Each question is a record: type (single_choice, multiple_choice, likert, text, ranking, matrix), prompt, options (for choices), required, branch_rules, and optional weight for scoring. Store questions as ordered rows or JSON. The application interprets the schema; it never hard-codes a specific instrument.

Anonymity and respondent identity

Three common modes. Anonymous: store no user id or PII; use a hashed respondent_token (cookie/session fingerprint) only to prevent duplicate votes — never reversible to identity. Identified: link responses to user_id for longitudinal studies. Confidential: store user id separately from answers in different tables with strict access controls so analysts can't join them casually. Document the mode in the survey record; enforce it in storage and export APIs.

Duplicate vote prevention

Polls require at-most-once participation. Layer checks: authenticated users → unique index on (survey_id, user_id); anonymous → hash a stable token (signed cookie + survey salt) and unique index on (survey_id, respondent_token_hash). Rate-limit by IP as a backstop, not the sole guard (shared NAT). Return a friendly 'already voted' without revealing whether others voted.

Aggregation: sync counters vs query-on-read

For live polls, precomputed counters win: on each response, increment choice counts in an aggregate_options table or Redis hash — O(1) reads for dashboards. For complex surveys (cross-tabs, filters, open-text search), query normalized answers on demand or maintain a reporting projection. Never expose per-response rows in public live results; only publish totals.

Branching logic (skip logic)

Questions can show/skip based on earlier answers ('if satisfaction < 3, show follow-up'). Store rules as data referencing question ids. The client uses them for UX, but the server must re-evaluate during validation — only require/accept questions that should be visible given submitted answers. Skipped questions must not appear in required validation.

Real-time results

When show_live_results is enabled, push aggregate updates after each response via Laravel broadcasting (SurveyResultsUpdated on a public channel keyed by survey slug). Cache the latest totals in Redis; the broadcast payload carries counts only, never individual answers. Disable or delay live results until the poll closes if early votes would bias participation (bandwagon effect).

Survey lifecycle and quotas

Surveys move through draft → open → closed. Respect close_at, max response count, and per-segment quotas (e.g. stop accepting when 100 responses from '18–24' are reached). Check status and quota inside a transaction before insert to avoid race overshoot.

Most common misconception
Treating a poll's 'one vote per person' as client-side only, or storing anonymous tokens in plain text, are the classic survey-platform failures. Enforce participation rules and anonymity on the server with hashed tokens and database constraints.
Principle 3 — Different Media

The same ideas, four ways

One-line summary

A survey platform stores questions as a schema, collects validated responses with server-enforced anonymity and duplicate-vote rules, maintains aggregate counters for fast reporting, re-evaluates branching on the server, and optionally broadcasts live totals without leaking individual answers.

Analogy — the ballot box and tally board

The ballot is the question schema; the slot accepts one envelope per voter (duplicate prevention). Anonymous surveys use a numbered stub torn off and hashed — you can tell if someone already voted, but not who they are. The tally board on the wall shows running totals (precomputed aggregates), updated after each ballot is counted. Branching is like conditional pages on a paper form ('if you answered No, skip to question 5') — and the official counter re-checks those rules, not the voter.

Lesson 2 — Hash anonymous respondent tokens

function respondentTokenHash(Survey $survey, string $rawToken): string {
    return hash_hmac('sha256', $rawToken, $survey->anonymity_salt);
}
// store respondent_token_hash only — never the raw token or user_id when anonymous

Lesson 3 — Build validation from visible questions

public function rulesFor(Survey $survey, array $visible): array {
    $rules = [];
    foreach ($survey->questions as $q) {
        if (!in_array($q->id, $visible)) continue;
        $r = $q->required ? ['required'] : ['nullable'];
        $rules["answers.{$q->id}"] = match ($q->type) {
            'single_choice' => array_merge($r, ['string', Rule::in($q->optionIds())]),
            'multiple_choice' => array_merge($r, ['array'], ['answers.'.$q->id.'.*' => Rule::in($q->optionIds())]),
            'likert' => array_merge($r, ['integer', 'between:1,5']),
            default => $r,
        };
    }
    return $rules;
}

Lesson 4 — Submit with duplicate-vote guard

public function submit(Request $r, Survey $survey) {
    abort_if($survey->isClosed(), 403);
    $tokenHash = $this->respondentTokenHash($survey, $r->cookie('rs_token'));
    abort_if($survey->responses()->where('respondent_token_hash', $tokenHash)->exists(), 409);

    $visible = $this->resolveVisibleQuestions($survey, $r->input('answers', []));
    $data = $r->validate($this->rulesFor($survey, $visible));

    DB::transaction(function () use ($survey, $data, $tokenHash) {
        $survey->responses()->create(['answers' => $data['answers'], 'respondent_token_hash' => $tokenHash]);
        UpdateSurveyAggregates::dispatch($survey, $data['answers']); // queued
    });
}

Lesson 5 — Re-evaluate branching server-side

function resolveVisibleQuestions(Survey $survey, array $answers): array {
    return $survey->questions->filter(function ($q) use ($answers) {
        foreach ($q->branch_rules ?? [] as $rule) {
            $val = $answers[$rule['question_id']] ?? null;
            if (!$this->branchMatches($rule, $val)) return false;
        }
        return true;
    })->pluck('id')->all();
}

Lesson 6 — Increment aggregate counters

// aggregate_options: survey_id, question_id, option_id, count
foreach ($answers as $questionId => $choice) {
    foreach ((array) $choice as $optionId) {
        AggregateOption::updateOrCreate(
            compact('surveyId', 'questionId', 'optionId'),
            []
        )->increment('count');
    }
}

class SurveyResultsUpdated implements ShouldBroadcast {
    public function broadcastWith(): array {
        return ['totals' => $this->totals]; // counts/percentages only

    public function broadcastOn(): Channel {
        return new Channel('surveys.'.$this->slug);

Step 2 — Automaticity
Principle 5 — Test Yourself

Retrieval-practice quiz

Answer before revealing.

Q1. How do a poll and a survey differ in this architecture?
Same engine (question schema + responses + aggregates); polls are short, often public, with strict one-vote rules and immediate tallies. Surveys are longer, may allow partial completion, branching, and quotas. Model both with shared tables; differ by kind/status rules.
Q2. What three anonymity modes exist and how do they affect storage?
Anonymous: no user_id/PII, only hashed respondent_token for dedup. Identified: link response to user_id. Confidential: separate identity from answers with strict access controls. The survey record declares the mode; storage and exports must enforce it.
Q3. How do you prevent duplicate votes in an anonymous poll?
Issue a stable token (signed cookie), hash it with a survey-specific salt, store only the hash, and enforce a unique index on (survey_id, respondent_token_hash). Rate-limit by IP as backstop. Never rely on client-side checks alone.
Q4. Why precompute aggregate counters for live results?
O(1) reads for dashboards and websocket payloads; querying all responses per page view doesn't scale. Increment counters on each response (DB or Redis), broadcast totals only — never individual rows.
Q5. How must branching (skip) logic be handled on the server?
Re-evaluate visibility from submitted answers before validation; only require/accept questions that should show. Client branching is UX only — hidden questions must not be required or stored as answered.
Q6. What checks run before accepting a response?
Survey not closed (status/close_at), quota not exceeded, duplicate vote not detected, visible questions resolved, dynamic validation passes — ideally inside a transaction to avoid race overshoot on quotas.

Code drill — duplicate vote rejected (Pest)

it('rejects a second vote from the same anonymous token', function () {
    $survey = Survey::factory()->anonymous()->open()->create();
    $token = 'abc123';
    $hash = hash_hmac('sha256', $token, $survey->anonymity_salt);
    $survey->responses()->create(['respondent_token_hash' => $hash, 'answers' => ['q1' => 'a']]);

    $this->withCookie('rs_token', $token)
        ->postJson("/surveys/{$survey->slug}/respond", ['answers' => ['q1' => 'b']])
        ->assertStatus(409);
});

Principle 7 — Mix It Up

Interleaved quiz

Topics shuffled to train choosing the right idea.

M1. Live poll totals lag 30 seconds behind submissions. Likely cause?
Aggregates updated in a slow queued job or queried on-read instead of incrementing counters. Increment precomputed counters synchronously (or fast queue) and cache latest totals for broadcast.
M2. An 'anonymous' survey exports user emails alongside answers. Fix?
Anonymity mode violated — anonymous surveys must not store or join PII. Separate confidential identity if needed; enforce mode in schema, storage, and export APIs.
M3. Users report voting twice from the same phone on a public poll. Bug?
Duplicate prevention missing or client-only. Add hashed respondent_token with unique DB constraint; authenticated polls also unique on (survey_id, user_id).
M4. A follow-up question is required even when the branch should skip it. Bug?
Branch logic not re-evaluated server-side. Compute visible questions from answers before building validation rules.
M5. Early voters skew results because everyone sees live percentages. Mitigation?
Disable live results until close, or show totals without percentages, or delay broadcast — bandwagon bias is a product rule, not just a tech issue.
M6. Quota for 'Region: West' exceeded by 3 responses under concurrent load. Fix?
Check quota inside a transaction with row lock or atomic counter decrement; reject when quota reached. Race without transaction allows overshoot.
Principle 8 — Don't Stop

Overlearning plan

  • Build duplicate prevention two ways — IP-only vs hashed token + unique index — and watch IP-only fail on shared networks.
  • Break anonymity on purpose — store raw cookies or join user_id to answers in an 'anonymous' survey, then audit exports.
  • Write bypass tests first: second vote, hidden-question injection, closed survey submit, quota race.
  • Add multi-page surveys with partial saves, resume tokens, and progress percentage.
  • Stretch goals: cross-tab reporting, weighted scoring (NPS), survey logic builder UI, CSV/SPSS export, quota segments, scheduled open/close, and bias-aware live-result modes.

Reference — Glossary

Key terms

Survey schemaThe set of questions (as data) defining an instrument.
QuestionA typed prompt definition: type, options, required, branch rules, weight.
PollA short, often public survey with strict one-vote rules and live tallies.
ResponseA respondent's set of answers to a survey, stored keyed by question id.
Respondent tokenA stable client identifier hashed for anonymous duplicate-vote prevention.
Anonymity modeWhether responses link to identity: anonymous, identified, or confidential.
Branching logicRules showing or skipping questions based on earlier answers.
Aggregate countersPrecomputed per-option counts updated on each response for fast reads.
QuotaA cap on responses for a survey or segment (e.g. demographic bucket).
Live resultsReal-time broadcast of aggregate totals, not individual answers.
Skip logicServer-re-evaluated rules determining which questions are visible.
Reporting projectionDenormalized answer store derived from responses for analytics/export.
Bandwagon effectParticipation bias when voters see early results before casting their vote.
