Step 1 — Understanding
Principle 1 — Map of the System

How a multi-tenant SaaS fits together

Multi-tenancy means one running application serves many isolated customers (tenants) from shared infrastructure. The whole discipline reduces to one question repeated at every layer: "whose data is this, and how do I guarantee it never leaks to another tenant?"

Every request flows through the same pipeline. The job of a tenancy system is to identify the tenant early, bootstrap a tenant context, and then make every downstream component — DB, cache, files, queues, mail — automatically scoped to that tenant.

   ───────────►  │     TENANT       header / JWT claim           │

The four pieces you will keep returning to:

  • Tenancy model — row-level (shared DB + tenant_id), database-per-tenant, or schema-per-tenant. This single choice drives everything else.
  • Tenant identification — the middleware that turns an incoming request into "this is tenant X".
  • Context bootstrapping & isolation — making connections, cache, storage, and queries respect the active tenant automatically, not by hand.
  • Central concerns — the tenant registry, custom domains, billing/subscriptions, provisioning, and cross-tenant admin.
The mental model in one sentence
A tenancy package is mostly middleware that mutates global state (connections, config, cache prefix) for the lifetime of one request or one job, then cleans up. Get the lifecycle right and the rest is detail.
Principle 2 — Clear Explanations

The concepts, defined precisely

Tenant, central, and the two databases

A tenant is one customer account with its own users and data. The central database (sometimes "landlord") holds records that exist above any tenant: the tenants table, domains, plans, and subscriptions. A tenant database (or tenant rows) holds the application data that belongs to exactly one customer. The reason for splitting them: "which tenants exist" must be answerable before any tenant context is active.

The three isolation models

Row-level (single database). Every tenant-owned table carries a tenant_id column. Isolation is enforced in software via a global query scope. Cheapest to operate, easiest to query across tenants, but a single missing scope leaks data — isolation is only as strong as your discipline.

Database-per-tenant. Each tenant gets its own database; a middleware swaps the default connection per request. Strong isolation, simple per-tenant backup/restore and export, but migrations and connection management get heavier and "all tenants" analytics is awkward.

Schema-per-tenant. One database server, one schema (PostgreSQL) per tenant. A middle ground — strong-ish isolation with shared server resources. Postgres-specific and operationally fiddly.

Tenant identification

Turning a request into a tenant. Common strategies, often combined:

  • Subdomain — acme.app.com → tenant "acme". The default for most SaaS.
  • Custom domain — app.acme.com mapped in a domains table. Premium-tier feature.
  • Path — app.com/acme/.... Avoids wildcard DNS/TLS but pollutes every route.
  • Header / API key / JWT claim — for stateless APIs and machine-to-machine.

Context bootstrapping & teardown

Bootstrapping is everything that must change once the tenant is known: point the default DB connection at the tenant DB (or activate the row-level scope), prefix the cache store, root the filesystem disk under the tenant, and rebind tenant-specific config. Teardown reverses it. Teardown matters most in long-lived processes — Octane workers and queue workers reuse the same PHP process across many requests/jobs, so a forgotten reset means tenant B inherits tenant A's connection. This is the number-one source of cross-tenant bugs.

The bug that defines the domain
Cross-tenant data leakage. It happens when (a) a query runs without the tenant scope, or (b) context from a previous request/job is not torn down in a reused worker process. Treat both as security incidents, not bugs.

Provisioning

Creating a tenant is a workflow, not an INSERT: write the central record, create the database (DB-per-tenant) or seed baseline rows, run tenant migrations, create the admin user, register the domain, and start the subscription/trial. Long and failure-prone — it belongs in a queued, idempotent job.

Principle 3 — Different Media

The same ideas, four ways

Summary

A tenancy system identifies the tenant from the request, bootstraps a per-tenant context (DB, cache, storage), enforces isolation automatically, and tears the context down afterward. Pick the isolation model first; everything else follows from it.

Analogy — the apartment building

The building is your app; tenants are residents. Row-level is a shared loft where everyone's belongings sit in one room with name tags (tenant_id) — cheap, but mislabel one box and a stranger gets your stuff. Database-per-tenant gives each resident a locked apartment — strong walls, but the building manager needs a key ring (connection management) and must repair every unit separately (per-tenant migrations). The doorman is your identification middleware: he reads the name on the door (subdomain) and routes you to the right unit. The danger is a doorman who forgets to lock the previous apartment before opening the next — that is missing teardown in a reused worker.

Principle 4 — Short Lessons

Bite-sized micro-lessons

Each lesson is one idea you can hold in your head. Build mentally as you read; the code is illustrative, not copy-paste production.

Lesson 1 — Decide the isolation model before writing code

This is an architecture decision, not a coding one. Ask: How many tenants? How large each? Compliance requirements (data residency, "delete my data")? Need for cross-tenant reporting? Default for most B2B SaaS: database-per-tenant for isolation, or row-level when you expect thousands of small tenants and need easy aggregate analytics. Packages: stancl/tenancy (most popular, supports both) or spatie/laravel-multitenancy (lean, opinionated).

Lesson 2 — Identify the tenant in middleware, early

Identification must run before any tenant query. Resolve from the request, then initialize the tenant.

// routes/web.php — tenant routes grouped behind identification
Route::middleware([
    InitializeTenancyByDomain::class,   // resolve tenant from host
    PreventAccessFromCentralDomains::class,
])->group(function () {
    Route::get('/', DashboardController::class);
});

// Conceptually, the middleware does:
$tenant = Tenant::on('central')->whereHas('domains',
    fn ($q) => $q->where('domain', $request->getHost()))->firstOrFail();
tenancy()->initialize($tenant);   // bootstrap context (Lesson 3)

Lesson 3 — Bootstrapping swaps global state

Initializing a tenant runs a set of bootstrappers. Each mutates one global concern:

  • Database — set the default connection to the tenant DB (or fill the tenant connection's database name).
  • Cache — prefix every key with the tenant id so Cache::get('stats') can't collide.
  • Filesystem — root the local/public disk at storage/tenant{id}/.
  • Queue — inject the tenant id into every dispatched job's payload.
// Spatie style: a task implementing TenancyBootstrapper
public function bootstrap(Tenant $tenant): void {
    config(['cache.prefix' => 'tenant_'.$tenant->id]);
    Cache::forgetDriver(config('cache.default')); // rebuild with new prefix
}
public function revert(): void { /* restore central prefix */ }

Lesson 4 — Enforce row-level isolation with a global scope

In the row-level model, never hand-write where('tenant_id', …). Attach it once, globally, so it's impossible to forget.

trait BelongsToTenant {
    protected static function bootBelongsToTenant(): void {
        static::creating(fn ($m) => $m->tenant_id ??= tenant()->id);
        static::addGlobalScope('tenant', function (Builder $b) {
            if (tenancy()->initialized) {
                $b->where($b->getModel()->getTable().'.tenant_id', tenant()->id);
            }
        });
    }
}
Watch the gaps a global scope can't cover
whereHas on a non-scoped relation, raw queries, DB::table() (bypasses Eloquent entirely), and firstOrCreate race conditions. Add a unique index on (tenant_id, …) as a database-level backstop.

Lesson 5 — Migrations: central vs tenant

Keep two migration paths. database/migrations/ for central tables; database/migrations/tenant/ for per-tenant schema. Run tenant migrations across all tenants — and make new-tenant provisioning run them automatically.

php artisan migrate                 # central
php artisan tenants:migrate         # loop every tenant's DB (stancl)
php artisan tenants:migrate --tenants=acme   # one tenant

Lesson 6 — Make queues tenant-aware

A queued job runs later, in a different process, with no tenant context. You must carry the tenant id in the payload and re-initialize on the worker — and tear down after, because the worker process is reused for the next job (which may be another tenant or a central job).

// stancl tags jobs automatically when dispatched in tenant context,
// and a QueueTenancyBootstrapper re-initializes + reverts per job.
// Manual pattern if rolling your own:
class GenerateInvoice implements ShouldQueue {
    public function __construct(public int $tenantId, public int $invoiceId) {}
    public function handle(): void {
        tenancy()->initialize(Tenant::find($this->tenantId));
        try { /* ... tenant-scoped work ... */ }
        finally { tenancy()->end(); } // ALWAYS revert
    }
}

Lesson 7 — Isolate cache, storage, and mail too

Data isolation isn't only the DB. A shared cache key, a shared storage path, or a queued mail rendered in the wrong context all leak. Cache: tenant-prefixed (Lesson 3). Storage: tenant-rooted disk, and never serve another tenant's files via a guessable URL — generate signed/temporary URLs. Mail: render inside tenant context so branding, locale, and "from" address are correct.

Lesson 8 — Billing & subscriptions live centrally

Subscriptions belong to the tenant record in the central DB (you must bill a suspended tenant whose tenant DB you might have archived). Use Laravel Cashier (Stripe/Paddle) on the Tenant model. Gate access by subscription state in middleware: past-due → grace banner; canceled → read-only or lockout. Handle webhooks centrally (no tenant context) and map the customer back to a tenant.

Lesson 9 — Provision tenants with an idempotent job

class CreateTenant implements ShouldQueue {
    public function handle(): void {
        $tenant = Tenant::create([...]);          // central row
        $tenant->domains()->create(['domain'=>...]);
        $tenant->run(function () {                 // inside tenant context
            Artisan::call('migrate', ['--path'=>'database/migrations/tenant']);
            User::create([...]);                  // admin
            (new SeedDefaults)->__invoke();
        });
        $tenant->newSubscription('default', $priceId)->trialDays(14)->create();
    }
}

Lesson 10 — Test isolation as a first-class concern

Your most important test is not "feature works" — it's "tenant B cannot see tenant A's data". Write that test before the feature. (Code in Principle 5.)

Step 2 — Automaticity
Principle 5 — Test Yourself

Retrieval-practice quiz

Answer out loud or on paper before revealing. Recall effort is what builds memory — peeking early wastes the rep.

Q1. You choose database-per-tenant. Name the two operational costs you've just signed up for.
(1) Migrations now run ×N — every schema change loops over all tenant databases, and provisioning must migrate new tenants automatically. (2) Connection management + cross-tenant analytics become hard — "all tenants" reporting needs a loop or separate ETL, since there is no single table to aggregate.
Q2. In a row-level model, why is a global query scope safer than adding where('tenant_id', …) in each query?
Because isolation then can't be forgotten. A per-query filter relies on every developer remembering it on every query forever; one omission leaks data across tenants. A global scope applies automatically to all Eloquent queries on the model. (Still back it with a DB unique index and watch raw/DB::table queries the scope can't reach.)
Q3. Why does a queued job risk cross-tenant leakage even when your HTTP middleware is correct?
The job runs later in a separate, long-lived worker process with no inherited tenant context. You must serialize the tenant id into the payload and re-initialize tenancy in handle() — and tear it down in a finally block, because the same worker process then picks up the next job (possibly a different tenant or a central job).
Q4. Where do subscription/billing records live, and why there rather than in the tenant DB?
In the central database, on the tenant record (via Cashier). You must be able to bill, suspend, or read the state of a tenant whose tenant database may be archived, locked, or not yet provisioned. Billing exists "above" the tenant, and Stripe/Paddle webhooks arrive with no tenant context.
Q5. Beyond the database, name three resources that must be isolated per tenant, and the failure if you don't.
(1) Cache — without a per-tenant key prefix, Cache::get('stats') returns another tenant's value. (2) Filesystem/storage — without a tenant-rooted disk + signed URLs, files collide or are guessable across tenants. (3) Queued mail / rendered views — rendered in the wrong context they ship the wrong branding, locale, or data. (Also: full-text/search indexes, broadcasting channels.)
Q6. Write (in words) the single most important test of a multi-tenant system.
Create tenant A with a record; create tenant B; initialize B's context; assert the model query returns zero of A's records (and that fetching A's record by id from B throws/404s). Isolation is the property under test, not the feature.

Code drill — the isolation test (Pest)

it('does not leak records across tenants', function () {
    $a = Tenant::factory()->create();
    $b = Tenant::factory()->create();

    tenancy()->initialize($a);
    Project::factory()->count(3)->create();
    tenancy()->end();

    tenancy()->initialize($b);
    expect(Project::count())->toBe(0);          // B sees none of A's
    tenancy()->end();
});

Principle 7 — Mix It Up

Interleaved quiz

Topics deliberately shuffled — identification, isolation, queues, billing, testing jump around so you practise choosing the right idea, not pattern-matching a section.

M1. A tenant on a custom domain app.acme.com reports they sometimes see a different company's dashboard after deploys. Octane is enabled. First hypothesis?
Missing teardown in the long-lived Octane worker — tenant context from a previous request bled into the next because the process is reused. Ensure tenancy is reverted between requests (Octane needs explicit context reset; verify the tenancy package's Octane integration/listener is registered).
M2. You need a nightly "total revenue across all tenants" report. Which isolation model makes this easy, which makes it painful, and what's the workaround for the painful one?
Easy: row-level (one query, group by tenant). Painful: database-per-tenant — no single table to aggregate. Workaround: loop tenants and accumulate, or run an ETL that copies metrics into a central reporting store/warehouse.
M3. DB::table('projects')->get() returns rows from all tenants in your row-level app despite the global scope. Why, and the fix?
DB::table() uses the query builder directly and bypasses Eloquent global scopes entirely. Fix: go through the Eloquent model, or if you must use the builder, add the tenant_id filter manually — and add a unique/composite index so the DB enforces tenant ownership where possible.
M4. New-tenant signup occasionally creates the central record but leaves the tenant with no schema. Where's the design flaw?
Provisioning isn't atomic/idempotent. Steps after the central INSERT (create DB, migrate, seed admin) failed and weren't retried. Move provisioning into a single queued, idempotent job with retries; mark the tenant status=provisioning until all steps succeed, and don't route traffic to it until active.
M5. A Stripe webhook arrives. There's no subdomain, no tenant context. How do you apply it to the right tenant?
Handle webhooks on the central domain/connection. Map the event's Stripe customer id back to the tenant record (Cashier stores it centrally), update the subscription state there. No tenant context needed — billing state lives centrally on purpose.
M6. When would you deliberately choose row-level over database-per-tenant despite weaker isolation?
Many small tenants (thousands+), low per-tenant data volume, heavy need for cross-tenant analytics, and a small ops team — the ×N migration/connection/backup overhead of DB-per-tenant would dominate, and the cost of strong isolation isn't justified. Compensate with global scopes + DB constraints + isolation tests.
Principle 8 — Don't Stop

Overlearning plan

Past the point of "I get it" is where durable, fast recall is built. Pick from these to keep the system live in your hands:

  • Build it twice. Implement a tiny tenant app once with stancl/tenancy (DB-per-tenant) and once with spatie/laravel-multitenancy (row-level). Feeling both clarifies the trade-offs better than any table.
  • Break it on purpose. Remove the queue tenancy bootstrapper and watch a job write to the wrong tenant. Remove a global scope and watch the isolation test fail. Knowing the failure mode beats memorising the fix.
  • Write the isolation test suite first on your next real feature. Make "no cross-tenant leak" a CI gate.
  • Read the source. Read stancl/tenancy's bootstrappers and Octane/queue listeners. It's the canonical reference for lifecycle correctness.
  • Teach it. Write a 1-page internal doc: "How tenancy works in our app, and the three ways to leak data." If you can write it cleanly, you own it.
  • Stretch goals: custom-domain onboarding with automated TLS; per-tenant feature flags & plan limits; tenant data export ("download all my data"); tenant deletion/GDPR erasure; per-tenant Scout/search indexes; cross-tenant impersonation for support with an audit log.
