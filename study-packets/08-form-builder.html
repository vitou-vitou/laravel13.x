Step 1 — Understanding
Principle 1 — Map of the System

How a form builder fits together

A form builder is a data-driven schema that generates UI, validation, and storage at runtime. The pressure: forms are defined by users (not developers), so fields are data; the same schema must render the form, validate submissions server-side, and shape how responses are stored and reported.

User (form creator)
 └─ Form (schema = ordered fields as JSON/rows)
     ├─ Fields (type, label, required, options, rules, logic)
     └─ Submissions (answers keyed by field)
          └─ used for export, aggregation, webhooks
 Public respondent ──► fills + submits ──► validated against schema

The defining action — a submission — is validated against the schema, not hard-coded rules:

Respondent submits answers
   │
   ▼ load form schema (fields + rules)
   ▼ build validation rules dynamically from schema
   ▼ validate ; on fail → errors keyed by field
   │ on pass
   ▼ store Submission (answers) ; fire webhooks/notifications (queued)
   ▼ spam checks (honeypot, rate limit, captcha)
  • Schema model — fields as data (type, options, required, validation, conditional logic).
  • Dynamic validation — translate the schema into Laravel validation rules at request time.
  • Submission storage — JSON answers vs normalized, and how that affects reporting.
  • Logic & protection — conditional show/hide and spam defences.
The mental model in one sentence
A form builder is a runtime interpreter of a user-defined schema: one schema drives rendering, validation, and storage. The hard part is generating trustworthy server-side validation from data.
Principle 2 — Clear Explanations

The concepts, defined precisely

Fields are data, not code

Each field is a record: type (text/email/select/checkbox/file/number/date), label, required, options (for choices), and validation rules. Store fields as ordered rows or a JSON array on the form. The application interprets this schema; it never hard-codes a specific form.

Generating validation from the schema

The critical security point: never trust the client. On submit, load the schema and build Laravel validation rules dynamically — e.g. a required email field becomes ['answers.fld_3' => 'required|email']. Validation must derive from the server-side schema, not from client-sent rules, or attackers bypass it.

Submission storage: JSON vs normalized

Two patterns. JSON answers: one submissions row with an answers JSON keyed by field id — simple, flexible, but harder to aggregate/query per field. Normalized: an answers table (submission_id, field_id, value) — easier per-field reporting, more rows/joins. JSON is the common default; add normalization or a reporting projection when analytics demand it.

Field identity and schema evolution

Forms change over time. Key answers by a stable field id (not label or position), so renaming a field or reordering doesn't orphan past responses. Consider versioning the schema so a submission records which schema version it answered.

Conditional logic

Fields can show/hide based on earlier answers ('if country = US, show state'). Store logic as data (conditions referencing field ids). The client uses it for UX, but the server must re-evaluate it during validation — only require/accept fields that should be visible given the submitted answers.

File uploads

File fields need validated mime/size, storage on a disk (S3), and the stored path/URL recorded in the answer. Stream large files, scan if needed, and use signed/temporary URLs so responses aren't publicly guessable.

Spam & abuse protection

Public forms attract bots. Layer defences: a honeypot field (hidden; if filled, drop), rate limiting per IP, optional CAPTCHA, and timing checks. Keep them server-side; client-only checks are trivially bypassed.

Most common misconception
Trusting client-sent validation rules (or validating only in the browser) is the classic form-builder vulnerability. Always rebuild and run validation from the server-side schema on every submission.
Principle 3 — Different Media

The same ideas, four ways

One-line summary

A form builder stores forms as field schemas, interprets them at runtime to render UI and to generate server-side validation, then stores submissions keyed by stable field ids — with conditional logic re-evaluated and spam filtered on the server.

Analogy — the recipe card

A form is a recipe card (the schema) that the kitchen (server) follows to both cook (render) and inspect the dish (validate). You'd never let a diner hand you their own inspection checklist (client-sent rules) — the kitchen keeps the authoritative recipe. Ingredients are labelled by a fixed code, not their position on the card, so reordering the recipe doesn't ruin past meals (stable field ids). Some steps only apply under conditions ('if vegetarian, skip the stock') — and the kitchen re-checks those conditions itself.

Principle 4 — Short Lessons

Bite-sized micro-lessons

Each lesson is one idea. Code illustrative.

Lesson 1 — Model fields as data

// fields: id, form_id, type, label, required, options(json), rules(json), logic(json), position
class Field extends Model {
    protected $casts = ['options'=>'array','rules'=>'array','logic'=>'array','required'=>'boolean'];
}

Lesson 2 — Build validation from the schema

public function rulesFor(Form $form, array $visible): array {
    $rules = [];
    foreach ($form->fields as $f) {
        if (!in_array($f->id, $visible)) continue; // hidden by logic
        $r = [];
        if ($f->required) $r[] = 'required';
        if ($f->type === 'email') $r[] = 'email';
        if ($f->type === 'number') $r[] = 'numeric';
        $rules["answers.{$f->id}"] = array_merge($r, $f->rules ?? []);
    }
    return $rules;
}

Lesson 3 — Validate then store

public function submit(Request $r, Form $form) {
    $visible = $this->resolveVisibleFields($form, $r->input('answers', []));
    $data = $r->validate($this->rulesFor($form, $visible));
    $form->submissions()->create(['answers'=>$data['answers']]);
    DispatchWebhooks::dispatch($form, $data['answers']); // queued
}

Lesson 4 — Re-evaluate conditional logic server-side

function resolveVisibleFields(Form $form, array $answers): array {
    return $form->fields->filter(function ($f) use ($answers) {
        foreach ($f->logic ?? [] as $cond) {
            if (($answers[$cond['field']] ?? null) !== $cond['equals']) return false;
        }
        return true;
    })->pluck('id')->all();
}

Step 2 — Automaticity
Principle 5 — Test Yourself

Retrieval-practice quiz

Answer before revealing.

Q1. Why are form fields modelled as data rather than code?
Forms are defined by users, not developers. Storing fields as records/JSON lets one application interpret any form — rendering, validating, and storing — without code changes per form.
Q2. Where must validation rules come from, and why?
From the server-side schema, rebuilt on each submission — never from client-sent rules or browser-only checks. Otherwise an attacker simply omits or alters the rules and bypasses validation.
Q3. Compare JSON vs normalized submission storage.
JSON answers (one row, keyed by field id) is simple and flexible but harder to aggregate per field. A normalized answers table eases per-field reporting at the cost of rows/joins. JSON is the default; add normalization/projection for analytics.
Q4. Why key answers by a stable field id instead of label or position?
So renaming a field or reordering the form doesn't orphan or misalign past responses. The id is the durable identity; labels and positions change.
Q5. How must conditional (show/hide) logic be handled on the server?
Re-evaluate it during validation: determine which fields should be visible given the submitted answers, and only require/accept those. Client-side logic is UX only and untrusted.
Q6. Name three server-side spam defences for a public form.
Honeypot field (drop if filled), per-IP rate limiting, and optional CAPTCHA/timing checks. All enforced server-side; client-only checks are trivially bypassed.
Q7. How are file uploads handled in answers?
Validate mime/size, store on a disk (e.g. S3), record the path in the answer, and serve later via temporary signed URLs so responses aren't publicly guessable.

Code drill — server validation can't be bypassed (Pest)

it('rejects a submission missing a required field regardless of client', function () {
    $form = Form::factory()->withField(['type'=>'email','required'=>true,'id'=>'fld_e'])->create();
    $this->postJson("/f/{$form->id}", ['answers'=>[]])
        ->assertStatus(422)->assertJsonValidationErrors('answers.fld_e');
});

Principle 7 — Mix It Up

Interleaved quiz

Topics shuffled to train choosing the right idea.

M1. A pentester submits invalid data by editing the page and it saves. Root cause?
Validation was client-side or trusted client-sent rules. Rebuild validation from the server schema on every submission so the browser can't be the authority.
M2. After a form owner renamed a field, old responses show blanks for it. Why and fix?
Answers were keyed by label/position, which changed. Key answers by a stable field id (and version the schema) so renames/reorders don't orphan data.
M3. Analytics on a single question are painfully slow with JSON storage. Options?
Query JSON paths with an index/generated column, or maintain a normalized answers table / reporting projection populated on submit for per-field aggregation.
M4. A 'state' field is required even when the user picked a non-US country. Bug?
Conditional logic not re-evaluated server-side — the field should be hidden (and not required) given the answers. Compute visible fields from submitted answers before validating.
M5. Bots are flooding a public form. First, cheapest defences?
Add a honeypot field (drop if filled) and per-IP rate limiting; escalate to CAPTCHA/timing checks if needed. All server-side.
M6. When is a normalized answers table worth the extra complexity?
When per-field reporting/aggregation/filtering is a core feature at scale, where JSON-path queries become too slow. For most forms, JSON answers (optionally with a projection) are simpler and enough.
Principle 8 — Don't Stop

Overlearning plan

  • Build validation two ways — trusting client rules vs rebuilding from schema — and exploit the first to feel why the second is mandatory.
  • Break field identity on purpose — key answers by label, rename a field, watch responses orphan.
  • Write the bypass tests first: missing required, hidden-field injection, honeypot tripped.
  • Add multi-page forms with partial saves and resume tokens.
  • Stretch goals: answer-piping/personalization, calculated fields, webhook + Zapier delivery, response analytics dashboard, and schema versioning per submission.
