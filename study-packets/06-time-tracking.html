Step 1 — Understanding
Principle 1 — Map of the System

How a time tracker fits together

A time tracker is a log of time intervals attached to billable work, then rolled up and approved. The pressure: a timer can be running (no end yet), entries must not silently overlap, billing depends on rates and rounding rules, and timesheets flow through approval before invoicing.

User
 ├─ Projects ─► Tasks (billable?, rate)
 ├─ Time Entries (start, end|null, duration, billable, rate)
 │     └─ running entry = end IS NULL
 ├─ Timesheet (a period's entries) ─► approval workflow
 └─ Reports (hours by project/user/day, billable totals)

The defining action — stopping a timer — closes the interval and computes billable value:

Stop timer ──► PATCH /entries/{id}/stop
   │ entry.end = now()
   ▼ duration = end - start  (apply rounding rule)
   ▼ billable_amount = round(duration) * rate  (if billable)
   │
   └─► update running-timer state; recompute day/period totals (cached)
  • Time entry model — intervals with a nullable end (running) tied to a task/project.
  • Integrity rules — one running timer per user, overlap detection, rounding.
  • Billing — billable flag, rate resolution, rounding policy → amount.
  • Approval & reporting — timesheet submission/approval and aggregate hours.
The mental model in one sentence
A time tracker is an interval log with billing rules and an approval gate. The hard parts are running-timer state, overlap/rounding correctness, and fast aggregation — not basic CRUD.
Principle 2 — Clear Explanations

The concepts, defined precisely

Time entries and the running timer

A time entry has started_at and a nullable ended_at. A running timer is simply an entry with ended_at = null. Duration is computed (ended_at - started_at), or now() - started_at while running. Don't store a separate 'is_running' flag — the null end is the single source of truth.

One running timer per user

Most trackers enforce one active timer per user. Starting a new timer auto-stops the previous. Enforce with a guard (and ideally a partial unique index on user_id where ended_at is null) so concurrent start requests can't create two running entries.

Overlap detection

Manually-entered entries can overlap (logging 9–11 and 10–12). Decide policy: reject overlaps, or allow and flag. Detect with an interval query: two intervals overlap when start < other.end AND end > other.start. This must run on create/edit, scoped to the user.

Rounding rules

Billing often rounds durations — to the nearest 6 minutes (0.1h), 15 minutes, or up to a minimum. Rounding is a policy applied at calculation time, not stored raw. Keep the true duration and apply rounding when computing billable amounts so you can change the rule later.

Rate resolution

The billable rate can come from the task, the project, the user, or a client contract — with a precedence order. Resolve it explicitly (task rate ?? project rate ?? user default) at entry time, and snapshot it onto the entry so later rate changes don't rewrite history.

Timesheet approval workflow

Entries for a period form a timesheet that a user submits; a manager approves or rejects (back to draft). Lock approved entries against edits. This is a small state machine (draft → submitted → approved/rejected) gating downstream invoicing.

Reporting

Reports aggregate hours by project, user, day, and billable status. Aggregate in SQL (sum, group by) over an indexed entries table; cache heavy rollups. For a 'running total today' use a cheap query against today's entries plus the live timer delta computed in the app.

Most common misconception
Storing duration as a mutable field that you increment is the classic bug — clock skew, missed stops, and double counting corrupt it. Derive duration from started_at/ended_at; keep the running timer as a null end.
Principle 3 — Different Media

The same ideas, four ways

One-line summary

A time tracker logs intervals (running = null end), enforces one active timer and no silent overlaps, applies rounding and rate rules to compute billable amounts, and gates timesheets through approval before reporting/invoicing.

Analogy — the parking meter

A time entry is a parking session: it starts when you arrive and the duration is just arrival-to-departure — you wouldn't keep a running 'minutes' counter that could drift, you'd read the two clock stamps. You can only occupy one space at a time (one running timer); double-parking (overlap) is flagged. The fee rounds to the next block and uses the posted rate at the time you parked (rate snapshot). At month end the lot manager reviews and approves the tickets before billing.

Principle 4 — Short Lessons

Bite-sized micro-lessons

Each lesson is one idea. Code illustrative.

Lesson 1 — Running timer = null end

// time_entries: user_id, task_id, started_at, ended_at(nullable), rate, billable
class TimeEntry extends Model {
    protected $casts = ['started_at'=>'datetime','ended_at'=>'datetime','billable'=>'boolean'];
    public function getRunningAttribute(): bool { return $this->ended_at === null; }
    public function durationSeconds(): int {
        return ($this->ended_at ?? now())->diffInSeconds($this->started_at);
    }
}

Lesson 2 — Enforce one active timer

public function start(StartTimer $r): TimeEntry {
    DB::transaction(function () use ($r) {
        TimeEntry::where('user_id',$r->user()->id)->whereNull('ended_at')
            ->update(['ended_at'=>now()]); // auto-stop previous
    });
    return TimeEntry::create([...]);
}
// + partial unique index: (user_id) WHERE ended_at IS NULL

Lesson 3 — Detect overlaps on manual entry

$overlap = TimeEntry::where('user_id',$userId)
    ->where('started_at','<',$newEnd)
    ->where('ended_at','>',$newStart)
    ->exists();
abort_if($overlap, 422, 'Time entry overlaps an existing one.');

Lesson 4 — Apply rounding at calc time

function roundedHours(int $seconds, int $incrementMin = 6): float {
    $hours = $seconds / 3600;
    $step  = $incrementMin / 60;
    return ceil($hours / $step) * $step; // round up to nearest increment
}

Lesson 5 — Resolve and snapshot the rate

$rate = $task->rate ?? $project->rate ?? $user->default_rate;
$entry->rate = $rate;          // snapshot so future rate changes don't rewrite history
$entry->billable = $task->billable;

Lesson 6 — Timesheet approval state machine

enum Sheet: string { case Draft='draft'; case Submitted='submitted';
    case Approved='approved'; case Rejected='rejected'; }
// submit: draft->submitted; approve: submitted->approved (lock entries);
// reject: submitted->rejected->draft (editable again)

Lesson 7 — Aggregate in SQL

TimeEntry::whereBetween('started_at',[$from,$to])
    ->selectRaw('project_id, SUM(TIMESTAMPDIFF(SECOND, started_at, COALESCE(ended_at, NOW()))) secs')
    ->groupBy('project_id')->get();
Step 2 — Automaticity
Principle 5 — Test Yourself

Retrieval-practice quiz

Answer before revealing.

Q1. How is a running timer represented, and why not an is_running flag?
As a time entry with ended_at = null. The null end is the single source of truth; a separate flag can desync from reality (e.g. set true while end is also set). Duration derives from started_at to ended_at (or now).
Q2. How do you guarantee only one active timer per user under concurrent requests?
Auto-stop the previous running entry inside a transaction on start, backed by a partial unique index on user_id WHERE ended_at IS NULL so the database rejects a second concurrent running row.
Q3. State the interval-overlap condition.
Two intervals overlap when newStart < existingEnd AND newEnd > existingStart. Run this query scoped to the user on create/edit of manual entries.
Q4. Why apply rounding at calculation time rather than storing rounded duration?
Rounding is a billing policy that may change. Keep the true duration; apply the rounding rule when computing billable amounts so history isn't corrupted and the rule can be revised.
Q5. Why snapshot the rate onto the entry?
So later changes to task/project/user rates don't retroactively rewrite past billable amounts. Resolve precedence (task ?? project ?? user) once and store it on the entry.
Q6. What does the timesheet approval workflow protect?
It gates entries before invoicing: draft → submitted → approved/rejected. Approved entries are locked against edits, ensuring billed time isn't silently changed.
Q7. How should 'hours this week by project' be computed?
In SQL with SUM of (ended_at - started_at), COALESCE-ing now() for running entries, grouped by project, over an indexed entries table — not by looping entries in PHP.

Principle 7 — Mix It Up

Interleaved quiz

Topics shuffled to train choosing the right idea.

M1. A user's totals are inflated; they have two running timers. How did it happen and how prevent it?
Concurrent start requests both created entries with null end. Prevent with a transactional auto-stop on start plus a partial unique index on user_id WHERE ended_at IS NULL.
M2. Finance changed the project rate and last month's invoice totals changed too. Bug?
Rate was read live instead of snapshotted. Store the resolved rate on each entry at creation so historical amounts are immutable.
M3. Reports are slow over a year of entries. Two fixes?
(1) Aggregate in SQL (SUM/GROUP BY) with indexes on (user_id/project_id, started_at). (2) Cache or precompute period rollups; don't loop entries in PHP.
M4. Billable amounts are a few cents off expectations. Likely cause?
Rounding applied inconsistently or to stored rounded durations. Keep true durations and apply one rounding policy at calculation time (e.g. ceil to nearest 6 min).
M5. An approved timesheet got edited and re-billed. What was missing?
A lock on approved entries. The approval state machine must forbid edits once approved (reopen only via reject → draft).
M6. When is an event-log model (start/stop events) better than start/end columns?
When you need a full audit trail of pauses/resumes and tamper evidence. It's more complex but reconstructs any state; for most apps start/end columns with derived duration are simpler and sufficient.
Principle 8 — Don't Stop

Overlearning plan

  • Build duration two ways — derived from start/end vs a stored counter — and reproduce the drift/double-count bug.
  • Break the single-timer rule on purpose — remove the unique index, hammer the start endpoint concurrently, watch two running timers appear.
  • Write rounding + overlap tests first across edge cases (zero-length, midnight crossing, back-to-back).
  • Add idle detection — prompt to discard idle minutes on long-running timers.
  • Stretch goals: invoicing from approved timesheets, per-client contracts and rate cards, time-off/leave integration, and a 'pomodoro' running-timer UI fed by a single source-of-truth query.
