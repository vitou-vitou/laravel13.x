Step 1 — Understanding
Principle 1 — Map of the System

How an invoicing app fits together

An invoicing app is a money document that must total correctly, number sequentially, and track payment. The pressure: monetary math must be exact (never floats), invoice numbers must be gapless for compliance, an invoice is immutable once issued, and partial/late payments change its status.

Customer
 └─ Invoice (number, status, currency, totals)
     ├─ Line Items (qty, unit_price, tax_rate, line_total)
     ├─ Payments (amount, method, date) ─► partial/full
     └─ PDF snapshot
 Recurring profile ─► generates Invoices on schedule

The defining action — issuing an invoice — assigns a number, freezes totals, and starts the dunning clock:

Issue invoice ──► assign next gapless number (atomic)
   │ compute line totals + tax + grand total (integer cents)
   ▼ status: draft → sent ; due_at set
   ▼ render + store PDF snapshot ; email customer (queued)
   └─► scheduled job flips to overdue past due_at if unpaid
  • Document model — invoice + line items, totals computed and frozen on issue.
  • Money & tax — integer minor units, exact tax rounding per line vs total.
  • Numbering & status — atomic gapless sequence; draft→sent→paid/overdue lifecycle.
  • Payments & recurring — partial payments, reconciliation, and scheduled generation.
The mental model in one sentence
An invoice is an immutable, exactly-totalled, sequentially-numbered money document. The hard parts are precise money/tax math and gapless numbering under concurrency — not the form.
Principle 2 — Clear Explanations

The concepts, defined precisely

Never use floats for money

Floating point can't represent 0.10 exactly, so sums drift by cents. Store money as integer minor units (cents) or use a decimal/money library (e.g. brick/money). All arithmetic happens in integers; format to currency only at display.

Line items and totals

An invoice hasMany line items (description, quantity, unit price, tax rate). Line total = qty × unit price; tax = line total × rate. The invoice subtotal, tax, and grand total are sums of line values, computed once and stored on the invoice at issue so the document is self-consistent forever.

Per-line vs per-invoice tax rounding

Rounding tax per line then summing can differ by a cent from taxing the subtotal. Jurisdictions specify which is correct. Pick a documented rule (commonly round each line, then sum) and apply it consistently — this is a frequent source of off-by-a-cent disputes.

Gapless invoice numbering

Many tax regimes require sequential, gapless invoice numbers. You cannot use the DB auto-increment id (gaps from deletes/rollbacks) or compute max()+1 (races). Use an atomic counter: a dedicated sequence row locked in a transaction (SELECT ... FOR UPDATE) so two concurrent issues can't claim the same or skip a number.

Immutability after issue

Once sent, an invoice must not change — you issue a credit note or a new invoice instead of editing. Lock line items and totals after issue; only status and payments evolve. This is both a legal and trust requirement.

Status lifecycle & dunning

draft → sent → (partially_paid) → paid, or sent → overdue past the due date if unpaid. A scheduled job flips unpaid sent invoices to overdue and can send reminders (dunning). Status is derived from payments + due date, recorded explicitly for querying.

Payments & recurring

A payment records amount/method/date; an invoice is paid when payments sum to the total (handle partials and overpayment). Recurring billing stores a profile (interval, template) and a scheduled job generates the next invoice — idempotently, so a double-run doesn't bill twice.

Most common misconception
Using floats for money or max(number)+1 for numbering are the two classic invoicing bugs — one drifts cents, the other duplicates/skips legally-required sequence. Use integer cents and an atomic counter.
Principle 3 — Different Media

The same ideas, four ways

One-line summary

An invoicing app builds immutable money documents from line items, totals them in integer cents with a defined tax-rounding rule, assigns gapless numbers atomically, and tracks payments to drive status.

Analogy — the cash register receipt

An issued invoice is a printed receipt: once it prints you can't scribble on it — a mistake means a refund slip (credit note), not an edit. The register counts in whole pennies, never fuzzy fractions. Receipts come off the roll in strict order with no missing numbers — you can't have receipt 1002 without 1001. And the till is balanced by matching payments to receipts.

Principle 4 — Short Lessons

Bite-sized micro-lessons

Each lesson is one idea. Code illustrative.

Lesson 1 — Money in integer cents

// store amounts as integers (cents); cast helper for display
class Invoice extends Model {
    protected $casts = ['total_cents'=>'integer'];
    public function totalFormatted(): string {
        return number_format($this->total_cents / 100, 2);
    }
}

Lesson 2 — Compute and freeze totals on issue

$subtotal = $items->sum(fn($i) => $i->qty * $i->unit_price_cents);
$tax = $items->sum(fn($i) => intdiv($i->qty * $i->unit_price_cents * $i->tax_bp, 10000));
$invoice->update(['subtotal_cents'=>$subtotal,'tax_cents'=>$tax,
    'total_cents'=>$subtotal + $tax]); // frozen at issue

Lesson 3 — Atomic gapless number

$number = DB::transaction(function () {
    $seq = Sequence::where('key','invoice')->lockForUpdate()->first();
    $seq->increment('value');
    return $seq->value; // formatted e.g. INV-2026-000123
});

Lesson 4 — Lock the document after issue

public function update(Request $r, Invoice $invoice) {
    abort_if($invoice->status !== 'draft', 422, 'Issued invoices are immutable; issue a credit note.');
    // ... only drafts editable
}

Lesson 5 — Mark overdue on schedule

// routes/console.php
Schedule::call(function () {
    Invoice::where('status','sent')->where('due_at','<',now())
        ->update(['status'=>'overdue']);
})->dailyAt('02:00');

Lesson 6 — Record payments, derive paid

$invoice->payments()->create(['amount_cents'=>$amt,'method'=>$m,'paid_at'=>now()]);
$paid = $invoice->payments()->sum('amount_cents');
$invoice->update(['status' => $paid >= $invoice->total_cents ? 'paid'
    : ($paid > 0 ? 'partially_paid' : $invoice->status)]);

Lesson 7 — Recurring generation, idempotent

// guard against double-run: unique (profile_id, period)
RecurringProfile::due()->each(function ($p) {
    Invoice::firstOrCreate(
        ['profile_id'=>$p->id,'period'=>$p->currentPeriod()],
        $p->buildInvoiceAttributes(),
    );
});
Step 2 — Automaticity
Principle 5 — Test Yourself

Retrieval-practice quiz

Answer before revealing.

Q1. Why must money never be stored as a float?
Floats can't represent decimals like 0.10 exactly, so sums drift by cents over many operations. Store integer minor units (cents) or use a money library; format to currency only at display.
Q2. Why can't you use the DB auto-increment id or max(number)+1 for invoice numbers?
Auto-increment leaves gaps (deletes/rollbacks), violating gapless-sequence tax rules; max()+1 races under concurrency, producing duplicates or skips. Use an atomic locked sequence row.
Q3. Describe the atomic gapless-numbering technique.
In a transaction, SELECT the sequence row FOR UPDATE (locking it), increment, and commit. Concurrent issuers block until the lock releases, so each gets a unique consecutive number with no gaps.
Q4. Why are totals computed once and stored on the invoice at issue?
So the document is self-consistent and immutable: later price/tax changes can't alter a sent invoice. The stored subtotal/tax/total are the legal record.
Q5. What's the per-line vs per-invoice tax-rounding issue?
Rounding tax on each line then summing can differ by a cent from taxing the subtotal once. Jurisdictions mandate one; pick and document a consistent rule to avoid off-by-a-cent disputes.
Q6. How should you correct an error on an issued invoice?
Don't edit it — issue a credit note or a corrective invoice. Issued invoices are immutable for legal and trust reasons; only status and payments may change.
Q7. How is recurring billing made safe against double runs?
Idempotent generation: firstOrCreate keyed on (profile_id, period) with a unique constraint, so a re-run of the scheduler can't create two invoices for the same period.

Code drill — gapless under concurrency (Pest)

it('never issues duplicate invoice numbers concurrently', function () {
    $numbers = collect(range(1,20))->map(fn() => nextInvoiceNumber()); // simulate parallel
    expect($numbers->unique()->count())->toBe(20)
        ->and($numbers->max() - $numbers->min())->toBe(19); // no gaps
});

Principle 7 — Mix It Up

Interleaved quiz

Topics shuffled to train choosing the right idea.

M1. Customers report totals occasionally off by a cent. Two likely causes?
(1) Float money arithmetic drifting. (2) Inconsistent tax rounding (per-line vs per-invoice). Fix: integer cents everywhere and one documented rounding rule.
M2. An auditor finds invoice numbers 1001, 1003, 1004 — 1002 missing. How did it happen and how prevent?
A number was claimed then the transaction rolled back, or numbers come from auto-increment with a deleted row. Prevent by claiming the number only on successful issue via a locked sequence committed atomically with the invoice.
M3. Two staff issue invoices at the same instant and get the same number. Fix?
Replace max()+1 with SELECT ... FOR UPDATE on a sequence row inside a transaction so concurrent issuers serialize and each gets a unique consecutive number.
M4. A sent invoice was edited to change an amount. What should the system have done?
Blocked the edit (immutable once sent) and required a credit note / corrective invoice. Only drafts are editable.
M5. The recurring scheduler ran twice after a deploy and billed some customers twice. Root cause?
Non-idempotent generation. Use firstOrCreate keyed on (profile_id, period) with a unique index so re-runs are no-ops.
M6. When would you store money as a decimal column instead of integer cents?
When using a money library/DB DECIMAL with controlled scale and you need fractional minor units (e.g. unit prices at 4 dp for some currencies/commodities). Either is fine if arithmetic is exact and rounding is explicit — the cardinal rule is no binary floats.
