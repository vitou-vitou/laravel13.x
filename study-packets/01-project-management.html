Step 1 — Understanding
Principle 1 — Map of the System

How a Kanban app fits together

A project-management app is a hierarchy of orderable things that many people edit at once. The entire design pressure comes from two facts: cards must keep a stable, user-defined order, and several users drag them around simultaneously. Get ordering and concurrency right and the rest is CRUD.

The canonical hierarchy:

Workspace
 └─ Project (members, roles)
     └─ Board (a Kanban view)
         └─ Column / List  ("To Do", "In Progress", "Done")
             └─ Card / Issue  (title, assignees, labels, position)
                 ├─ Comments
                 ├─ Checklist items
                 ├─ Attachments
                 └─ Activity log entries

The request lifecycle for the one action that matters most — moving a card:

User drags card  ──►  PATCH /cards/{id}/move {column_id, after_id}
                         │
                         ▼
          Policy: can user edit this board?
                         │
                         ▼
          Compute new position (fractional / lexorank)
                         │
                         ▼
          Persist column_id + position  (single UPDATE)
                         │
             ┌───────────┴───────────┐
             ▼                       ▼
     Broadcast CardMoved        Write ActivityLog
     to board channel           (queued)
             ▼
     Other users' boards update live

The four subsystems you keep returning to:

  • Domain model — workspace → project → board → column → card, plus labels, comments, attachments.
  • Ordering — how a card knows it sits between two others, cheaply, under concurrent moves.
  • Authorization — project membership and roles decide who can see/edit what.
  • Real-time + activity — broadcasting changes to other viewers and recording an audit trail.
The mental model in one sentence
A Kanban app is an ordered tree edited concurrently. The defining engineering problem is card ordering under simultaneous drag-and-drop — solve that with fractional/lexorank positions and you avoid full-column renumbering and most race conditions.
Principle 2 — Clear Explanations

The concepts, defined precisely

The core data model

A board belongs to a project and hasMany columns; a column hasMany cards. A card belongs to a column and carries a position. Cards relate to users through an assignees pivot and to labels through a pivot. This is a straightforward relational tree — the subtlety is entirely in position.

Why integer positions hurt

The naive approach gives each card an integer 1,2,3… Dropping a card at the top means renumbering every card below it — many writes, and two concurrent moves easily produce duplicate or conflicting numbers. It does not scale and races badly.

Fractional / lexorank ordering

Instead, store a position that lets you insert between two neighbours without touching anyone else. Two common schemes: (1) fractional float — new position = midpoint of the neighbours' positions (e.g. between 1.0 and 2.0 → 1.5); (2) lexorank — a base-N string (e.g. "aaa", "aam", "aaz") where you generate a string lexicographically between two others. A move becomes a single UPDATE of one row. Periodic rebalancing handles precision exhaustion.

Authorization model

Access is driven by project membership with a role (owner/admin/member/viewer). A Laravel Policy on the Board/Card models checks the acting user's role on the parent project. Centralize it: every board/card query is scoped to projects the user belongs to, and every mutating route runs through the policy.

Real-time collaboration

When one user moves a card, others watching the board should see it move. Laravel broadcasting (Reverb/Pusher/Ably) pushes a CardMoved event to a private channel named per board (e.g. board.42); the channel authorization callback re-uses the same membership check. The frontend listens and reorders locally.

Optimistic UI vs server truth

The drag feels instant because the frontend moves the card immediately (optimistic update) then sends the request. If the server rejects it (lost permission, stale board) the UI rolls back. The server is the source of truth for final position; the broadcast reconciles everyone.

Activity log / audit trail

Every meaningful change (card created, moved, assigned, commented) appends an immutable activity row: actor, verb, subject, and a small JSON payload. Write it from model events or explicitly, and queue it so it never slows the user's action. The feed is just a paginated query of this table.

Comments, mentions, notifications

Comments belongTo a card and a user. An @mention parses usernames, links them, and fires a Laravel Notification (database + mail/broadcast) to the mentioned user. Notifications are queued.

Most common misconception
The number-one mistake is integer positions with renumber-on-insert: it generates O(n) writes per move and races under concurrency. Use fractional or lexorank positions so a move is a single-row update.
Principle 3 — Different Media

The same ideas, four ways

One-line summary

A Kanban app is an ordered tree of cards in columns, edited by many users at once; fractional/lexorank positions make moves cheap, policies guard access, and broadcasting keeps everyone's board in sync.

Analogy — the shared whiteboard

Picture a physical whiteboard with sticky notes in columns. Many people stand at it moving notes. Integer positions are like numbering every note and rewriting all the numbers whenever one moves — chaos when two hands move at once. Fractional positions are like saying "put it halfway between these two" — you only touch the one note. The policy is the room's lock: only team members get in. Broadcasting is everyone seeing the same board update the instant a note moves.

Principle 4 — Short Lessons

Bite-sized micro-lessons

Each lesson is one idea. Code is illustrative, not copy-paste production.

Lesson 1 — Model the tree with clear relationships

final class Card extends Model {
    protected $casts = ['position' => 'float', 'meta' => 'array'];
    public function column(): BelongsTo { return $this->belongsTo(Column::class); }
    public function assignees(): BelongsToMany { return $this->belongsToMany(User::class, 'card_user'); }
    public function labels(): BelongsToMany { return $this->belongsToMany(Label::class); }
    public function comments(): HasMany { return $this->hasMany(Comment::class); }
}

Lesson 2 — Order with fractional positions

Compute the midpoint of the neighbours at the drop target. New top = first/2; new bottom = last+1; between = (prev+next)/2.

public function positionBetween(?float $prev, ?float $next): float {
    if ($prev === null && $next === null) return 1.0;
    if ($prev === null) return $next / 2;
    if ($next === null) return $prev + 1;
    return ($prev + $next) / 2;
}

Lesson 3 — Move = one authorized UPDATE

public function move(MoveCardRequest $r, Card $card) {
    $this->authorize('update', $card);              // policy
    $pos = $this->positionBetween($r->prevPos, $r->nextPos);
    $card->update(['column_id' => $r->column_id, 'position' => $pos]);
    broadcast(new CardMoved($card))->toOthers();    // skip the actor
    return new CardResource($card);
}

Lesson 4 — Guard with a policy + scoped queries

class CardPolicy {
    public function update(User $u, Card $card): bool {
        return $card->column->board->project
            ->members()->whereKey($u->id)
            ->wherePivotIn('role', ['owner','admin','member'])->exists();
    }
}

Lesson 5 — Broadcast on a private board channel

// routes/channels.php
Broadcast::channel('board.{board}', function ($user, Board $board) {
    return $board->project->members()->whereKey($user->id)->exists();
});

class CardMoved implements ShouldBroadcast {
    public function __construct(public Card $card) {}
    public function broadcastOn(): PrivateChannel {
        return new PrivateChannel('board.'.$this->card->column->board_id);
    }
}

Lesson 6 — Queue the activity log so it never blocks

// fire-and-forget audit trail
Activity::queue(
    actor: $r->user(),
    verb: 'moved_card',
    subject: $card,
    payload: ['to_column' => $card->column_id],
);  // dispatched to a queue; the user's move already returned

Lesson 7 — Rebalance when precision runs out

After many midpoint inserts, floats converge (1.5, 1.25, 1.125…). When the gap between neighbours falls below a threshold, run a queued job that rewrites the whole column's positions to clean integers (1,2,3…). Rare, cheap, invisible to users.

Step 2 — Automaticity
Principle 5 — Test Yourself

Retrieval-practice quiz

Answer before revealing — recall effort builds memory.

Q1. Why are integer positions with renumber-on-insert a poor ordering strategy?
Inserting at the top renumbers every card below (O(n) writes), and two concurrent moves can produce duplicate or conflicting numbers. It neither scales nor handles concurrency.
Q2. How does a fractional position make a move cheap?
You set the moved card's position to the midpoint of its two new neighbours — a single-row UPDATE that touches no other card. Order is preserved by the numeric value.
Q3. What is the failure mode of fractional positions, and the fix?
Repeated midpoint inserts exhaust float precision (1.5 → 1.25 → 1.125 …). Fix: when the neighbour gap is below a threshold, run a queued rebalance job that rewrites the column's positions to clean integers.
Q4. Where does authorization live for moving a card, and what does it check?
In a Policy (e.g. CardPolicy@update) invoked by the controller. It checks that the user is a member of the card's board's project with a role that permits editing (owner/admin/member, not viewer).
Q5. How do other users see a card move in real time, and how is the channel secured?
The server broadcasts a CardMoved event to a private channel named per board (board.{id}). The channel authorization callback re-uses the project-membership check, so only members receive it. Use ->toOthers() to skip the actor who already updated optimistically.
Q6. Why queue the activity-log write instead of doing it inline?
The audit entry isn't needed for the user's action to succeed; queuing it keeps the move response fast and decouples the feed from the hot path. The feed is then a paginated read of the activity table.
Q7. What is optimistic UI here, and what must the client handle?
The frontend moves the card immediately, then sends the request. It must roll back if the server rejects (lost permission, stale state) and reconcile with the authoritative broadcast/response.

Code drill — concurrent move safety (Pest)

it('keeps order stable when two cards move into the same gap', function () {
    [$a,$b,$c] = makeCards([1.0, 2.0, 3.0]);
    // two users both drop between A and B
    moveBetween($b, prev:1.0, next:2.0); // -> 1.5
    moveBetween($c, prev:1.0, next:2.0); // -> 1.25 (recomputed from current neighbours)
    expect(orderedTitles())->toBe(['A','C','B']); // deterministic, no duplicates
});

Principle 7 — Mix It Up

Interleaved quiz

Topics shuffled so you practise choosing the right idea, not pattern-matching a section.

M1. A user reports cards occasionally jump to the wrong order after a teammate drags at the same moment. Likely cause?
Positions computed from stale client-sent neighbour values, or integer renumbering racing. Recompute the midpoint from current DB neighbours inside the move handler (ideally in a transaction), and prefer fractional/lexorank so concurrent inserts don't collide.
M2. Board updates stop appearing live for one user but moves still save. Where do you look first?
Broadcasting/channel auth, not the database. Check the private channel authorization for board.{id} (membership callback), the websocket connection (Reverb/Pusher), and that CardMoved implements ShouldBroadcast and is dispatched.
M3. The activity feed is slow and the board feels laggy on busy projects. Two fixes?
(1) Queue activity writes so they're off the move hot path. (2) Paginate/limit and index the activity table (board_id, created_at); consider a separate read model. Don't eager-load the whole history.
M4. After months, one column's cards refuse to reorder cleanly. Diagnosis?
Float precision exhausted from repeated midpoint inserts — neighbour gap underflowed. Run the queued rebalance job to rewrite that column's positions to clean integers; lower the rebalance threshold.
M5. A viewer-role user can move cards via the API though the UI hides the control. Flaw?
Authorization enforced only in the UI. The move route must call the Policy server-side; never rely on hidden frontend controls. Add a policy test for the viewer role.
M6. When would lexorank be worth its extra complexity over fractional floats?
Very long-lived columns with extremely frequent reordering where float precision would exhaust quickly, or when you want human-inspectable, collision-resistant string ranks and controlled rebalancing. For most apps, fractional floats + occasional rebalance are simpler and sufficient.
Principle 8 — Don't Stop

Overlearning plan

  • Build the move endpoint twice — once with fractional floats, once with a lexorank library. Feeling both teaches the trade-off better than the table.
  • Break it on purpose — switch to integer renumbering and reproduce the concurrency bug; then watch fractional positions fix it.
  • Write the authorization tests first on your next feature: viewer cannot move, non-member cannot subscribe to the channel.
  • Add real-time presence — show who's viewing the board (presence channels) and live cursors.
  • Stretch goals: swimlanes, WIP limits per column, card dependencies, full-text search (Scout), saved filters, and an undo built on the activity log.

Key terms

BoardA Kanban view belonging to a project; has many columns.
Column / ListAn ordered bucket of cards (e.g. To Do, Done).
Card / IssueA unit of work with title, assignees, labels, and a position.
PositionThe orderable value (float or lexorank string) that places a card among its siblings.
Fractional indexingOrdering by inserting at the numeric midpoint between neighbours.
LexorankString-based rank scheme allowing insertion between two ranks lexicographically.
RebalancePeriodic rewrite of a column's positions to restore spacing/precision.
PolicyLaravel authorization class deciding if a user may act on a model.
Private channelA broadcast channel requiring server-side authorization to subscribe.
Optimistic UIApplying a change on the client before the server confirms, rolling back on failure.
Activity logImmutable audit trail of actor/verb/subject/payload rows powering the feed.
Presence channelA broadcast channel that also tracks who is currently subscribed.
