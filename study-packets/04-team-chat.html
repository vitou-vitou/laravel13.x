Step 1 — Understanding
Principle 1 — Map of the System

How a chat app fits together

A chat app is a real-time fan-out problem over an append-only message log. The pressure: one message must reach every channel member instantly, history must paginate cleanly while new messages keep arriving, and each user needs an accurate unread count. Persistence is easy; delivery and counting are hard.

Workspace
 ├─ Users
 ├─ Channels (public / private)  ── members (pivot, last_read_at)
 │   └─ Messages (append-only, ordered)
 │       ├─ Reactions
 │       ├─ Thread replies
 │       └─ Attachments
 └─ Direct Messages (a channel of 2+, no name)

The defining action — sending a message — persists, then fans out over WebSockets:

User sends ──► POST /channels/{id}/messages
   │ authorize (member?)
   ▼
 Persist Message (gets id + created_at)   ◄ source of truth
   │
   ▼ broadcast MessageSent to presence channel chat.{id}
   │     (all connected members receive instantly via Reverb)
   ▼
 Queue: push notifications to offline members, bump unread
  • Data model — workspaces, channels (incl. DMs as nameless channels), append-only messages, membership pivot.
  • Real-time delivery — broadcasting MessageSent to a per-channel channel via Reverb/Pusher.
  • Ordering & pagination — cursor (keyset) pagination so infinite scroll is stable as new messages arrive.
  • Unread & presence — last_read_at per member for counts; presence channels for who's online.
The mental model in one sentence
A chat app is fan-out delivery over an append-only log. The hard parts are real-time delivery to all members, stable cursor pagination, and accurate per-user unread counts — not storing the message.
Principle 2 — Clear Explanations

The concepts, defined precisely

Channels, DMs, and membership

A channel groups messages and has members through a pivot. A direct message is just a channel with no name and a fixed membership — modelling DMs as channels avoids a parallel message table. The pivot row per member carries last_read_at, the key to unread counts and read receipts.

Messages are append-only

Messages are an immutable log: you insert, rarely update (edits keep an edited_at), and soft-delete. Order by (id) or (created_at, id). Treating the table as a log — not mutable rows — is what makes pagination and real-time consistent.

Real-time delivery via broadcasting

When a message is saved, you broadcast(new MessageSent($message)) to a channel named chat.{channelId}. Laravel Reverb (a first-party WebSocket server) or Pusher/Ably delivers it to every connected member's browser. The HTTP response and the broadcast are separate paths — the sender sees their own message from the response (or echo), others from the broadcast.

Why cursor pagination, not offset

Offset pagination (LIMIT 50 OFFSET 100) breaks in chat: new messages arriving shift every offset, causing duplicates or skips, and large offsets are slow. Cursor / keyset pagination (where id < :lastSeenId order by id desc limit 50) is stable regardless of inserts and uses the index directly. Essential for infinite scroll.

Unread counts

Per member, unread = messages in the channel newer than their last_read_at. Don't recount the whole table on every request — maintain a counter or compute with an indexed count where created_at > last_read_at. Mark-as-read updates last_read_at when the user views the channel.

Read receipts & presence

Read receipts derive from each member's last_read_at vs a message's timestamp. Presence (who's online/typing) uses a presence channel — a broadcast channel that tracks subscribers; typing indicators are ephemeral client events, never persisted.

Notifications & fan-out to the offline

Connected users get the WebSocket broadcast. Offline members need push/email — dispatched from a queued job after the message saves, respecting mute settings. Separate the two paths: real-time for present users, queued notifications for absent ones.

Most common misconception
Offset pagination is the classic chat bug: as messages arrive, offsets shift and users see duplicates or gaps. Use cursor/keyset pagination keyed on message id.
Principle 3 — Different Media

The same ideas, four ways

One-line summary

Chat persists an append-only message log, broadcasts each new message to all channel members over WebSockets, paginates history by cursor, and tracks unread via each member's last_read_at.

Analogy — the bulletin board with a town crier

The message log is a bulletin board where notices are only ever pinned (append-only), never reordered. The broadcast is a town crier who, the instant a notice goes up, shouts it to everyone currently in the square (connected members) — but people who went home (offline) get a letter later (queued notification). Unread is each person remembering the time they last read the board; anything newer is unread. Cursor pagination is reading the board from a remembered notice downward, so newly pinned notices at the top never make you lose your place.

Lesson 2 — Send = persist then broadcast

public function store(SendMessage $r, Channel $channel) {
    $this->authorize('post', $channel);
    $msg = $channel->messages()->create([
        'user_id'=>$r->user()->id, 'body'=>$r->body,
    ]);
    broadcast(new MessageSent($msg))->toOthers();
    NotifyAbsentMembers::dispatch($msg);   // queued
    return new MessageResource($msg);
}

Lesson 3 — Broadcast on a presence channel

// routes/channels.php
Broadcast::channel('chat.{channel}', function ($user, Channel $channel) {
    return $channel->members()->whereKey($user->id)->exists()
        ? ['id'=>$user->id,'name'=>$user->name] // presence payload
        : false;
});

class MessageSent implements ShouldBroadcast {
    public function __construct(public Message $message) {}
    public function broadcastOn(): PresenceChannel {
        return new PresenceChannel('chat.'.$this->message->channel_id);
    }
}

Step 2 — Automaticity
Principle 5 — Test Yourself

Retrieval-practice quiz

Answer before revealing.

Q1. Why model direct messages as channels rather than a separate table?
A DM is structurally a channel with no name and fixed membership. Reusing channels avoids a parallel message table, duplicated pagination/broadcast logic, and special-casing — DMs and group chats share one code path.
Q2. Why is offset pagination wrong for chat history?
New messages arriving shift every offset, so paging produces duplicates or skipped messages, and deep offsets scan many rows. Cursor/keyset pagination (id < lastSeen) is stable under inserts and uses the index.
Q3. What are the two separate delivery paths for a new message?
(1) Real-time broadcast over WebSockets (Reverb/Pusher) to currently connected members. (2) A queued job sending push/email to offline members (respecting mutes). Present vs absent users are served differently.
Q4. How is a per-user unread count computed, and what makes it cheap?
Count messages in the channel newer than the member's last_read_at (from the pivot). An index on (channel_id, created_at) keeps it fast; mark-as-read updates last_read_at when the user views the channel.
Q5. Why are messages treated as append-only?
An immutable log makes ordering, cursor pagination, and real-time consistency simple. Edits keep an edited_at and deletes are soft — the sequence of ids/timestamps never changes underneath readers.
Q6. What kind of broadcast channel powers presence and why?
A presence channel — it tracks who is subscribed, so you get the online roster for free and can layer typing indicators. Channel auth returns a user payload for members, false for non-members.
Q7. Why should typing indicators not be persisted?
They're ephemeral, high-frequency, and worthless after the moment. Send them as client whispers/whisper events over the socket; persisting would hammer the DB for no value.

Code drill — stable cursor paging (Pest)

it('does not duplicate messages when new ones arrive mid-scroll', function () {
    $c = Channel::factory()->create();
    $first = makeMessages($c, 50);          // ids 1..50
    $page1 = paginate($c, cursor:null);     // newest 50
    makeMessages($c, 5);                      // ids 51..55 arrive now
    $page2 = paginate($c, cursor:$page1->last()->id); // older than page1
    expect($page2->pluck('id')->intersect($page1->pluck('id')))->toBeEmpty();
});

Principle 7 — Mix It Up

Interleaved quiz

Topics shuffled to train choosing the right idea.

M1. Users scrolling up sometimes see the same message twice. Cause and fix?
Offset pagination shifting as new messages arrive. Switch to cursor/keyset pagination keyed on message id (where id < lastSeenId order by id desc).
M2. Messages save fine but other members don't see them live. Where do you look first?
Broadcasting, not the DB. Check the WebSocket server (Reverb running), the channel authorization callback for chat.{id}, that MessageSent implements ShouldBroadcast, and the client Echo subscription.
M3. Unread badges are slow on busy workspaces. Two fixes?
(1) Index (channel_id, created_at) and count against last_read_at rather than scanning. (2) Maintain a denormalized unread counter per membership updated on send/read, or compute counts in a batched query.
M4. Offline users miss messages entirely — no push, no email. What's missing?
The second delivery path. Dispatch a queued job after save to notify absent members (push/email), mute-aware. Real-time broadcast only reaches connected clients.
M5. A non-member connected to a channel's socket and received messages. Security flaw?
Presence/private channel authorization not enforcing membership. The Broadcast::channel callback must return false for non-members; never trust the client's channel name alone.
M6. Typing indicators are flooding the database with writes. Redesign?
Don't persist them. Send as client whispers over the existing socket connection (ephemeral); they never touch the DB.
Principle 8 — Don't Stop

Overlearning plan

  • Build pagination twice — offset vs cursor — and reproduce the duplicate-message bug, then watch cursor fix it.
  • Break delivery on purpose — disable the queued notify job and confirm offline users get nothing; then restore it.
  • Write the membership/auth tests first: non-members can't subscribe, can't post, can't read.
  • Add threads — replies referencing a parent message id, with a per-thread unread.
  • Stretch goals: message search (Scout/Meilisearch), reactions, edit/delete with live sync, presence-based online roster, and horizontal scaling of Reverb behind a load balancer.
