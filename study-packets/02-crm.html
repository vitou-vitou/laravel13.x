Step 1 — Understanding
Principle 1 — Map of the System

How a sales CRM fits together

A CRM is a system of record for relationships and the deals that flow through stages. The design pressure: the pipeline must be configurable per team, every touch must be captured on a unified timeline, and managers need aggregated reporting across thousands of records without N+1 collapse.

Organization
 ├─ Users (sales reps, managers)
 ├─ Contacts ──┐
 ├─ Companies ─┤ (people & accounts)
 ├─ Pipeline
 │   └─ Stages (Lead → Qualified → Proposal → Won/Lost)
 ├─ Deals (value, stage, owner, contact)
 │   └─ Activities (calls, emails, notes, tasks)  ◄ the timeline
 └─ Custom Fields (per-entity, per-org)

The core action — moving a deal through the pipeline — drives stage history, forecasting, and notifications:

Rep drags deal Qualified → Proposal
        │
        ▼  authorize (owns deal / manager)
  Update deal.stage_id + record StageTransition (history)
        │
   ┌────┴─────┐
   ▼          ▼
 Activity   Recompute forecast (queued)
 timeline   + notify manager if value > threshold
  • Entity model — contacts, companies, deals, with custom fields per organization.
  • Configurable pipeline — stages are data, not enums, so each team defines its own.
  • Activity timeline — a polymorphic stream of every interaction on a deal/contact.
  • Assignment & reporting — routing leads to reps and aggregating pipeline value.
The mental model in one sentence
A CRM is a configurable pipeline over a relationship graph, with a unified activity timeline. The hard parts are user-defined stages/fields (data, not code) and fast aggregate reporting.
Principle 2 — Clear Explanations

The concepts, defined precisely

Contacts, companies, and deals

A contact is a person; a company is an account; a contact belongsTo a company. A deal (opportunity) belongsTo a contact/company and an owner, and carries a monetary value and a current stage. These three are the spine of every CRM.

Pipeline stages as data, not enums

Stages must be configurable per team, so model them as a stages table (name, order, probability%) belonging to a pipeline. A deal references stage_id. Never hard-code stages as a PHP enum — different orgs need different funnels, and admins reorder them at runtime.

Stage history & forecasting

Each move appends a stage transition row (deal, from, to, at, by). This history powers velocity metrics (time in stage) and forecasting: expected revenue = Σ(deal value × stage probability). Without history you can only see the current state, not the trend.

The activity timeline (polymorphic)

Calls, emails, notes, and tasks all appear on one timeline. Model an activities table with a polymorphic subject (subject_type, subject_id) so an activity can attach to a deal, contact, or company. The timeline is a single ordered query, not a union of many tables.

Custom fields

Orgs add their own fields (e.g. "industry", "lead source"). Two patterns: an EAV table (custom_field_values: entity, field_id, value) — flexible but query-heavy; or a JSON column (meta) with a field-definition table for the UI — simpler, queryable via JSON paths in modern MySQL/Postgres. JSON column is usually the right default.

Lead assignment

New leads route to reps by rule: round-robin, by territory, or by load. Implement as a queued job that picks the next owner; keep the algorithm in a service so it's testable and swappable.

Reporting without N+1

Dashboards aggregate: pipeline value by stage, deals won per rep, conversion rates. Use database aggregation (selectRaw('stage_id, sum(value)')->groupBy), eager-load relations for lists, and cache expensive rollups. Never loop deals in PHP to sum — push it to SQL.

Most common misconception
Hard-coding pipeline stages as an enum is the classic CRM mistake — it makes the product un-configurable. Stages, fields, and assignment rules must be data the admin controls.
Principle 3 — Different Media

The same ideas, four ways

One-line summary

A CRM tracks contacts, companies, and deals moving through configurable pipeline stages, records every interaction on a polymorphic timeline, and aggregates it all into reports.

Analogy — the sales relay race

A deal is a baton passed down a track of stages. Each lane (pipeline) can have a different number of hurdles because teams sell differently — so the track is built from movable pieces (data), not poured concrete (enums). The timeline is the race commentary: every call, email, and note logged in order. The forecast is the bookmaker's odds: each stage has a probability, and expected revenue weights deal value by how far down the track it is.

Bite-sized micro-lessons

Each lesson is one idea. Code illustrative, not production.

Lesson 1 — Stages are rows, deals reference them

Schema::create('stages', function (Blueprint $t) {
    $t->id();
    $t->foreignId('pipeline_id')->constrained();
    $t->string('name');
    $t->unsignedInteger('order');
    $t->unsignedTinyInteger('probability'); // 0-100 for forecasting
});
// deals.stage_id -> stages.id

Lesson 2 — Record every transition

public function moveStage(Deal $deal, Stage $to): void {
    $from = $deal->stage_id;
    $deal->update(['stage_id' => $to->id]);
    StageTransition::create([
        'deal_id'=>$deal->id,'from_stage_id'=>$from,
        'to_stage_id'=>$to->id,'moved_by'=>auth()->id(),
    ]);
}

Lesson 4 — Forecast in SQL, not PHP

$forecast = Deal::query()
    ->join('stages','stages.id','=','deals.stage_id')
    ->where('deals.status','open')
    ->selectRaw('SUM(deals.value * stages.probability / 100) as expected')
    ->value('expected');

Lesson 5 — Round-robin assignment as a service

class RoundRobinAssigner {
    public function next(Team $team): User {
        return $team->reps()
            ->orderBy('open_deals_count') // least loaded first
            ->first();
    }
}
// called from a queued AssignLead job

Step 2 — Automaticity
Principle 5 — Test Yourself

Retrieval-practice quiz

Answer before revealing.

Q1. Why model pipeline stages as a table rather than a PHP enum?
Stages must be configurable per team — different orgs have different funnels and reorder/add stages at runtime. An enum bakes them into code, making the product un-configurable and every change a deploy.
Q2. What does a stage-transition history table enable that a single stage_id cannot?
Trend and velocity analysis: time-in-stage, conversion rates between stages, and historical forecasting. A bare stage_id shows only the current state, not how the deal got there.
Q3. How does one timeline hold calls, notes, and emails for both deals and contacts?
A polymorphic activities table (subject_type, subject_id via morphTo). Any entity can have a timeline, and it's a single ordered query rather than a union across separate tables.
Q4. Give the forecasting formula and where it should run.
Expected revenue = Σ(deal.value × stage.probability/100) over open deals. Run it in SQL with SUM/JOIN/GROUP BY — never loop deals in PHP to sum.
Q5. When would you choose a JSON meta column over an EAV table for custom fields?
Default to JSON meta when fields are admin-defined and filtering is light — it's simpler and queryable via JSON paths. Reach for EAV only when you need heavy per-field reporting/filtering at scale.
Q6. A deal list page is slow with many rows. First fix?
Eliminate N+1 by eager-loading contact, owner, and stage (with select of only needed columns) and paginating. Then index foreign keys/status used for filtering.
Q7. Where does lead-assignment logic belong and why?
In a dedicated service (e.g. RoundRobinAssigner) called from a queued job — so the algorithm is testable, swappable (round-robin/territory/load), and off the request hot path.

Code drill — forecast aggregation (Pest)

it('weights pipeline value by stage probability', function () {
    $s50 = Stage::factory()->create(['probability'=>50]);
    $s100 = Stage::factory()->create(['probability'=>100]);
    Deal::factory()->create(['value'=>1000,'stage_id'=>$s50->id,'status'=>'open']);
    Deal::factory()->create(['value'=>2000,'stage_id'=>$s100->id,'status'=>'open']);
    expect(forecast())->toBe(2500.0); // 500 + 2000
});

Principle 7 — Mix It Up

Interleaved quiz

Topics shuffled to train choosing the right idea.

M1. A manager wants 'average days a deal spends in Proposal'. Which table answers it and how?
The stage-transition history: for each deal, diff timestamps of entering and leaving the Proposal stage, then average. Impossible with only the current stage_id.
M2. Reporting dashboard times out on a large org. Two architectural fixes?
(1) Aggregate in SQL with GROUP BY and proper indexes (stage_id, owner_id, status, created_at). (2) Cache/precompute rollups (scheduled job or materialized summary table) instead of recomputing per request.
M3. Admins ask to add a 'Renewal' stage between Won and Closed mid-quarter. What breaks if stages were an enum?
A code change + migration + deploy, and existing forecasts/history keyed to enum values get messy. With stages-as-data it's an insert and a reorder — no deploy.
M4. Sales wants to filter deals by a custom 'industry' field. You used a JSON meta column — how, and the caveat?
whereJsonContains('meta->industry', ...) (MySQL/Postgres JSON path). Caveat: JSON-path filters are slower and may not use ordinary indexes — add a generated/virtual column + index, or move hot filters to real columns.
M5. Two reps both 'claim' the same inbound lead simultaneously. How prevent double assignment?
Make assignment atomic: a queued job with a DB transaction + row lock (or a unique constraint on lead+owner / a status flag updated conditionally) so only the first claim wins.
M6. When is EAV worth its join cost over a JSON column?
When you need heavy, indexed per-field querying/reporting across many admin-defined fields at scale — EAV lets you index field values relationally, whereas JSON paths get expensive. For most CRMs, JSON wins on simplicity.
Principle 8 — Don't Stop

Overlearning plan

  • Build the pipeline twice — stages as enum vs stages as data — and feel why data wins the moment an admin reorders.
  • Break reporting on purpose — sum deals in PHP across 50k rows, watch it crawl, then move it to SQL aggregation.
  • Write velocity tests first on stage history before building the dashboard.
  • Add email integration — log inbound/outbound email as timeline activities (mailbox parsing or webhook).
  • Stretch goals: deal forecasting by close date, lead deduplication/merge, territory-based assignment, custom-field validation rules, and a saved-views/filters system.
